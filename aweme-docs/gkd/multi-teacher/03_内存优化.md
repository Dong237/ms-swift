# 多Teacher GKD 内存优化指南

> 本文档详细分析多Teacher GKD训练中的GPU内存分配、各优化手段的效果、配置边界与限制，并提供面向不同规模场景的推荐配置。

---

## 1. 内存模型总览

### 1.1 组件构成

多Teacher GKD训练中，GPU内存由以下组件占用：

| 类别 | 组件 | 内存占用 |
|------|------|---------|
| **持久占用** | Student模型参数 | 2 × P_s bytes (bf16) |
| | Student优化器 | 8 × P_s bytes (AdamW: params+m+v+grad) |
| | Teacher模型参数 | 2 × P_t × N_teachers bytes (bf16) |
| | 梯度 | 2 × P_s bytes (bf16) |
| **瞬态占用** | 激活值 | ~2-8 GB (取决于batch_size × seq_len) |
| | vLLM KV Cache | 可配置 (gpu_memory_utilization) |
| | ZeRO-3 Gather | 每次gather一层参数的完整副本 |

### 1.2 关键公式

**无分片时单GPU内存（bf16训练）：**

```
M_total = M_student + M_optimizer + M_gradients + M_teachers + M_activations

M_student    = 2 × P_s                    （模型参数，bf16）
M_optimizer  = 2 × P_s × 3                （AdamW: fp32 params + m + v ≈ 12 bytes/param，
                                            但ms-swift默认bf16优化器 ≈ 6 bytes/param）
M_gradients  = 2 × P_s                    （bf16梯度）
M_teachers   = 2 × P_t1 + 2 × P_t2 + ... （所有Teacher参数，bf16）
M_activations ≈ batch_size × seq_len × hidden × num_layers × 2 bytes
```

**ZeRO-N分片后单GPU内存：**

| ZeRO Stage | 分片内容 | 单GPU占用 |
|------------|---------|----------|
| ZeRO-0 | 无分片 | 全部 |
| ZeRO-1 | 优化器状态 | M_student + M_gradients + M_optimizer/N + M_teachers |
| ZeRO-2 | 优化器 + 梯度 | M_student + (M_gradients + M_optimizer)/N + M_teachers |
| ZeRO-3 | 优化器 + 梯度 + 参数 | (M_student + M_gradients + M_optimizer)/N + M_teachers |

> 注：以上仅对Student应用ZeRO。Teacher的ZeRO配置由 `--teacher_deepspeed` 独立控制。

## 2. 具体场景内存计算

### 2.1 场景A：7B Student + 2×8B Teacher（8卡 A100 80GB）

| 组件 | 总内存 | ZeRO-2 单GPU（8卡） | 说明 |
|------|--------|---------------------|------|
| Student参数 (7B) | 14 GB | 14 GB | ZeRO-2不分片参数 |
| Student优化器 (AdamW bf16) | 42 GB | 5.25 GB | 优化器分片 |
| Student梯度 | 14 GB | 1.75 GB | 梯度分片 |
| Teacher 1 (8B) | 16 GB | 16 GB | Teacher不分片 |
| Teacher 2 (8B) | 16 GB | 16 GB | Teacher不分片 |
| 激活值 | ~4 GB | ~4 GB | batch=4, seq=4096 |
| **合计** | | **~47 GB** | 可行 ✅ |

**加 `--offload_teacher_model true` 后：**

| 组件 | 单GPU |
|------|-------|
| Student + 优化器 + 梯度 | ~21 GB |
| Teacher（最多1个在GPU） | 16 GB |
| 激活值 | ~4 GB |
| **合计** | **~41 GB** ✅ |

### 2.2 场景B：7B Student + 2×30B Teacher（8卡 A100 80GB）

| 组件 | 总内存 | ZeRO-2 单GPU | 说明 |
|------|--------|-------------|------|
| Student参数 | 14 GB | 14 GB | |
| Student优化器 + 梯度 | 56 GB | 7 GB | 分片 |
| Teacher 1 (30B) | 60 GB | 60 GB | 不分片，**单卡放不下** ❌ |

**解决方案：Teacher也用DeepSpeed分片**

使用 `--teacher_deepspeed zero2`（注意：多Teacher不能用ZeRO-3）：

| 组件 | 总内存 | 单GPU（teacher ZeRO-2, 8卡） |
|------|--------|------------------------------|
| Student参数 | 14 GB | 14 GB |
| Student优化器 + 梯度 | 56 GB | 7 GB |
| Teacher 1 (30B) 参数 | 60 GB | 60 GB（ZeRO-2不分片参数） |

**ZeRO-2也放不下！** 因为ZeRO-2只分片优化器和梯度，参数不分片。Teacher是frozen的，没有优化器和梯度，所以ZeRO-2对Teacher几乎无效。

**真正的解决方案：`--offload_teacher_model true`**

```
单GPU内存 = Student(14) + Student优化器+梯度(7) + 1个Teacher(60) + 激活(4)
         = ~85 GB → 还是放不下 ❌
```

**终极方案：多节点 + offload**

16卡（2节点×8卡），Student ZeRO-3：

| 组件 | 单GPU（16卡 ZeRO-3） |
|------|---------------------|
| Student参数 | 0.875 GB |
| Student优化器 + 梯度 | 3.5 GB |
| Teacher (30B, offload=1个在GPU) | 60 GB |
| 激活值 | ~4 GB |
| **合计** | **~68.4 GB** ✅ |

### 2.3 场景C：1.7B Student + 2×8B Teacher（64卡 H20 96GB）

| 组件 | 单GPU（64卡 ZeRO-3） |
|------|---------------------|
| Student参数 (1.7B) | 0.05 GB |
| Student优化器 + 梯度 | 0.21 GB |
| Teacher 1 (8B) | 16 GB |
| Teacher 2 (8B) | 16 GB |
| 激活值 | ~2 GB |
| vLLM (colocate, 15%) | ~14 GB |
| **合计** | **~48 GB** ✅ 余量充足 |

这个场景用 colocate 模式即可，无需 server 模式。

## 3. 内存优化手段详解

### 3.1 `--offload_teacher_model true`（CPU offload between steps）

**原理：** 每次Teacher forward pass之前从CPU加载到GPU，结束后卸载回CPU。多Teacher时任意时刻只有1个Teacher在GPU上。

**效果：**

| 场景 | 无offload | 有offload |
|------|----------|----------|
| 2×8B Teacher | 32 GB | 16 GB（只有1个在GPU） |
| 3×30B Teacher | 180 GB（不可能） | 60 GB（只有1个在GPU） |

**性能开销：** CPU↔GPU传输，通常每次 ~1-3 秒（取决于模型大小和PCIe带宽）。相对于训练step时间（通常数十秒），开销较小。

**推荐：** 多Teacher训练时**始终建议开启**。

```bash
--offload_teacher_model true
```

### 3.2 `--teacher_deepspeed zero2`（Teacher分片）

**原理：** 使用DeepSpeed ZeRO-2对Teacher进行优化器和梯度分片。但Teacher是frozen的（无优化器、无梯度），所以 **ZeRO-2对frozen Teacher几乎无效**。

**ZeRO-2实际效果：** 主要好处是将Teacher纳入DeepSpeed的内存管理，允许与 `offload_teacher_model` 配合使用。参数本身不分片。

```bash
--teacher_deepspeed zero2
```

### 3.3 ZeRO-3 限制

**⚠️ 多Teacher时不能使用 `--teacher_deepspeed zero3`**

原因：ZeRO-3要求所有rank同步参与AllGather来收集参数。多Teacher按样本路由时，不同rank的micro-batch可能包含不同channel的样本，导致某些rank调用Teacher A而其他rank调用Teacher B，AllGather死锁。

设置 `--teacher_deepspeed zero3` + `--teacher_domain_map` 时会直接报错：

```
ValueError: ZeRO-3 for teacher models is incompatible with multi-teacher per-sample routing.
```

**单Teacher仍然可以用ZeRO-3**（所有rank都调用同一个Teacher，无死锁风险）。

### 3.4 Teacher去重（自动）

多个channel指向同一模型路径时，只加载一次：

```bash
--teacher_domain_map '{"math":"/model/A","code":"/model/A","writing":"/model/B"}'
# 实际加载2个Teacher（不是3个）
# math和code共享teacher[0]，writing用teacher[1]
```

### 3.5 vLLM Server 模式（间接释放内存）

vLLM **仅用于学生的on-policy生成**，与Teacher推理完全无关。但通过将vLLM引擎移到独立GPU，可以释放训练GPU上的内存给Teacher使用。

| 模式 | vLLM位置 | 训练GPU可用内存 |
|------|---------|---------------|
| colocate | 共享训练GPU | 80GB - vLLM占用(~15%) ≈ 68GB |
| server | 独立GPU | 完整80GB |

```bash
# server模式：2张GPU给vLLM，6张给训练
CUDA_VISIBLE_DEVICES=6,7 swift rollout --model ... --port 8000
CUDA_VISIBLE_DEVICES=0,1,2,3,4,5 swift rlhf ... --vllm_mode server
```

### 3.6 LoRA（减少Student内存）

LoRA冻结Student大部分参数，大幅减少优化器和梯度内存：

| 配置 | Student优化器内存 (7B) |
|------|---------------------|
| Full finetune | ~42 GB |
| LoRA rank=64 | ~0.5 GB |

```bash
--tuner_type lora --lora_rank 64 --lora_alpha 128
```

## 4. 推荐配置矩阵

### 4.1 按Teacher大小选择

| Teacher大小 | 推荐配置 |
|------------|---------|
| ≤ 8B | `--offload_teacher_model true`，无需特殊处理 |
| 8B - 30B | `--offload_teacher_model true` + student用ZeRO-3 + 考虑vLLM server模式 |
| > 30B | 必须多节点 + `--offload_teacher_model true` + student ZeRO-3 + vLLM server模式 |

### 4.2 按GPU数量选择

| GPU配置 | 推荐方案 |
|---------|---------|
| 单节点 8卡 A100 80GB | `--offload_teacher_model true` + `--deepspeed zero2`，Teacher≤30B可行 |
| 2节点 16卡 | student `--deepspeed zero3` + `--offload_teacher_model true`，Teacher≤60B |
| 8节点 64卡 | student `--deepspeed zero3` + `--offload_teacher_model true`，几乎无Teacher大小限制 |

### 4.3 决策树

```
你的Teacher有多大？
│
├── 小 (≤ 8B每个)
│   └── --offload_teacher_model true
│       └── vLLM模式：colocate即可
│           └── 单节点足够
│
├── 中 (8B - 30B每个)
│   ├── 单节点 8卡？
│   │   └── --offload_teacher_model true + --deepspeed zero3
│   │       └── vLLM模式：server（释放内存给Teacher）
│   │
│   └── 多节点可用？
│       └── --offload_teacher_model true + --deepspeed zero3
│           └── vLLM模式：server（独立节点）
│
└── 大 (> 30B每个)
    └── 必须多节点
        └── --offload_teacher_model true + --deepspeed zero3
            └── vLLM模式：server（独立节点）
                └── 2+训练节点分摊Student分片
```

## 5. 限制与边界

### 5.1 硬性限制

| 限制 | 原因 | 错误信息 |
|------|------|---------|
| Teacher不能用ZeRO-3（多Teacher时） | AllGather死锁 | `ValueError: ZeRO-3 for teacher models is incompatible...` |
| Teacher不能通过vLLM推理 | GKD需要完整logits，vLLM API不返回 | N/A（架构限制） |
| Megatron后端不支持多Teacher | 未实现 | `NotImplementedError` |
| Padding-free / Packing 不兼容 | channel是per-sample元数据，packing破坏对应关系 | N/A（行为不正确） |

### 5.2 软性限制

| 限制 | 影响 | 应对 |
|------|------|------|
| Liger kernel + per-teacher超参 | 自动回退到标准路径，内存效率降低 | 可接受，或禁用Liger |
| Liger kernel + 混合Teacher批次 | 自动回退到标准路径 | 按channel排序数据减少混合批次 |
| 混合Teacher批次效率 | 每个Teacher做子集forward，有额外tensor操作开销 | 按channel排序数据优化 |

### 5.3 每Teacher最大内存边界（单卡 80GB A100）

假设Student 7B + ZeRO-2 + 无vLLM：

```
可用 = 80GB - Student参数(14GB) - Student优化器+梯度分片(7GB) - 激活(4GB)
     = 55GB

单个Teacher上限（offload=true）= 55GB / 2bytes_per_param ≈ 27.5B参数
```

使用 ZeRO-3（16卡）+ LoRA：

```
可用 = 80GB - Student参数分片(0.875GB) - Student优化器+梯度分片(0.5GB) - 激活(4GB)
     ≈ 74.6GB

单个Teacher上限（offload=true）≈ 37B参数
```

## 6. vLLM 与Teacher的关系

一个常见的误解是 `--vllm_mode server` 会将Teacher部署到远程GPU。**这是不对的。**

### 6.1 架构事实

```
vLLM引擎 ──── 仅负责 ──→ 学生模型的on-policy文本生成
                            │
Teacher模型 ── 始终是 ──→ PyTorch模型，在训练进程内做forward pass
                            │
两者完全独立，通过不同代码路径运行
```

- `training_step()` 中：vLLM生成学生响应（`self._fast_infer()`）
- `compute_loss()` 中：Teacher做forward pass（`teacher_model(**model_inputs)`）

### 6.2 Server 模式如何间接帮助

Server 模式将 vLLM 引擎移到独立GPU/节点，训练GPU不再需要为 vLLM 预留内存（colocate模式下约占15-20%）。这部分释放的内存可以给Teacher使用。

| 配置 | 80GB A100训练GPU可用内存 |
|------|------------------------|
| 无vLLM（off-policy） | 80 GB |
| vLLM colocate | ~64-68 GB |
| vLLM server | 80 GB |

### 6.3 Server vs Colocate 选择

| 选colocate | 选server |
|-----------|----------|
| Teacher小（≤8B） | Teacher大（>8B） |
| 想要最大训练吞吐 | 内存是瓶颈 |
| 设置简单 | 有独立GPU/节点可用 |
| 延迟敏感 | 愿意用训练GPU换内存 |

## 7. 不支持的内存优化方案

### 7.1 Teacher作为vLLM API服务

GKD需要完整的teacher logits（整个词表的概率分布），标准vLLM/OpenAI API只返回生成的token和top-k log-probability。Qwen3的词表大小为151,936，每个token需要传输 ~600KB（bf16），网络开销极大。

### 7.2 量化Teacher

ms-swift支持4-bit/8-bit量化（`--quant_bits`），但没有 `--teacher_quant_bits` 独立参数。理论上可以预先量化Teacher检查点后加载，但未经多Teacher GKD测试验证。

### 7.3 预计算离线蒸馏

将Teacher logits预先计算并存储为tensor，训练时只加载存储的logits，完全不需要Teacher模型。这可以消除所有Teacher内存占用，但：
- 不支持on-policy（学生生成新响应后无法获取Teacher logits）
- 存储开销巨大（每token ~600KB × 数据量）
- 超出当前ms-swift GKD实现范围
